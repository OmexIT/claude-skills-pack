# Documentation style guide (default)

## Voice
- Write in plain language.
- Prefer short sentences.
- Use active voice.
- Avoid jargon; if you must use it, define it once.

## Structure
- Start with: what this is, who it’s for, and what you’ll learn.
- Use headings to make scanning easy.
- Put the “happy path” first, then edge cases.

## Code and examples
- Include at least one complete example for any API.
- Show realistic values (not foo/bar) when possible.
- Call out common errors and how to fix them.

## Consistency
- Use consistent names for the same concept.
- Keep capitalization consistent for product features.
- If a term is overloaded, rename it or disambiguate explicitly.

## Checklists
- A reader should know:
  - prerequisites
  - step-by-step instructions
  - how to validate success
  - how to troubleshoot
