# API-First: Spring Boot Endpoint Scaffold Skill

Turns an OpenAPI 3.1 fragment (or a short resource spec) into a complete, compilable Spring Boot 4 endpoint stack: controller, service, repository, DTO, mapper, validation, exception handler, OpenAPI spec file, slice tests, and integration tests.

**Reference files** (loaded on demand by the agent):
- `references/CONVENTIONS.md` -- response envelope, error shape, money, IDs, timestamps
- `templates/TEMPLATES.md` -- full code templates for every generated artifact

---

## Before You Start -- Superpowers Workflow

This skill generates a multi-file resource stack. Run it through the superpowers development workflow for reliable output.

1. **superpowers:brainstorming** -- recommended for new resources. Explore: what fields, what validation, what error cases, what pagination strategy, what status codes, what idempotency requirements. Skipped only if the OpenAPI fragment is already comprehensive.
2. **superpowers:writing-plans** -- produce a plan listing the files to generate, the validation rules per field, and the test cases. Review before generation.
3. **superpowers:using-git-worktrees** -- isolate so controller/service/repo/mapper changes don't collide with other feature work.
4. **superpowers:test-driven-development** -- write the `@WebMvcTest` slice test FIRST (asserting 201 on create, 400 on invalid, 404 on missing, idempotency-key replay), then the integration test with Testcontainers. Then implement.
5. **superpowers:dispatching-parallel-agents** -- controller, service, repository, mapper, and migration can be generated by independent parallel agents, each in its own worktree file. Merge back to the feature branch when all waves complete.
6. **superpowers:verification-before-completion** -- compile (`./gradlew compileJava` or `./mvnw compile`), run slice tests, run integration tests. Paste output. Hit the live endpoint with `curl` and paste the envelope response. Do not claim done without these.
7. **superpowers:requesting-code-review** -- before merge. Reviewer focuses on: error shape (RFC 9457), pagination cursor correctness, mapper null handling, and OpenAPI spec parity.

## 1. Conventions

All generated code follows the conventions in `references/CONVENTIONS.md`. Key rules:

- **Paths**: plural kebab-case nouns, versioned (`/api/v1/payment-links`)
- **Response envelope**: `{ data, meta: { requestId, timestamp, traceId } }`; list responses add `meta.pagination` with cursor fields
- **Errors**: RFC 9457 Problem Details with field-level error array
- **Money**: string-typed `BigDecimal`, never float (`"150.00"`)
- **IDs**: UUID v7 (external) or Hypersistence TSID (internal), always exposed as string
- **Timestamps**: `Instant` / `OffsetDateTime`, ISO 8601 UTC

See `references/CONVENTIONS.md` for full JSON shapes and examples.

## 3. Generated Artifacts

Full code templates for every artifact are in `templates/TEMPLATES.md`. The templates cover:

| Artifact | Key patterns |
|----------|-------------|
| Controller | `@RestController`, `@Validated`, Idempotency-Key header, `ApiResponse` envelope, `@Operation` annotations |
| Service | `@Transactional`, idempotency store integration, TSID generation, cursor pagination delegation |
| DTOs | Java records with Jakarta validation (`@NotNull`, `@Positive`, `@Size`, `@Future`) |
| Mapper | MapStruct `@Mapper`, `toDomain` / `toResponse`, Money formatting expression |
| Repository | Spring Data JDBC, cursor-based pagination query with `pageSize + 1` pattern |
| Exception handler | `@ControllerAdvice` / `@ExceptionHandler`, RFC 9457 `ProblemDetail` |
| Slice test | `@WebMvcTest`, 201 on valid create, 400 on invalid input, `MockBean` for service and mapper |
| OpenAPI spec | 3.1 fragment with `$ref` schemas, Idempotency-Key parameter, response envelope |

**Minimal controller example** (full versions in `templates/TEMPLATES.md`):

```java
@RestController
@RequestMapping("/api/v1/payment-links")
@Validated
@RequiredArgsConstructor
@Tag(name = "Payment Links")
public class PaymentLinkController {

    private final PaymentLinkService service;
    private final PaymentLinkMapper mapper;

    @PostMapping
    @Operation(summary = "Create a payment link")
    public ResponseEntity<ApiResponse<PaymentLinkResponse>> create(
            @RequestHeader("Idempotency-Key") String idempotencyKey,
            @Valid @RequestBody PaymentLinkRequest request) {
        var link = service.create(mapper.toDomain(request), idempotencyKey);
        return ResponseEntity.status(201).body(ApiResponse.of(mapper.toResponse(link)));
    }
}
```

Refer to `templates/TEMPLATES.md` for the complete, copy-ready code for every artifact.

## 5. Anti-patterns (never generate)

- Manual DTO-to-Entity conversion -- always MapStruct
- `RestTemplate` in generated clients -- use `RestClient` (Spring 6.1+)
- Field injection (`@Autowired` on fields) -- constructor only
- `String.format` -- use `.formatted()` (Java 15+)
- Returning entities directly -- always map to response DTOs
- `double`/`float` for money -- always `BigDecimal`
- `java.util.Date` -- always `Instant` / `OffsetDateTime`
- `@Transactional(readOnly = false)` explicit -- omit; readOnly=false is default
- Multiple responsibilities in one controller class -- split per resource
- Exception handling inside controllers -- delegate to `@ControllerAdvice`
- Exposing auto-increment integer IDs -- always UUID/TSID string
